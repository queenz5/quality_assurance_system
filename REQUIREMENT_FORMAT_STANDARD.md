# 需求文档格式标准

## 现有格式规范（基于 qa_system/data/requirements）

### 文件命名规范
```
{模块名}/REQ-XXX_{需求标题}.md
```
示例：
- `基础组件/REQ-001_Icon 图标组件开发.md`
- `表单组件/REQ-002_Input 输入框组件.md`

### 文档结构
```markdown
# REQ-001: Icon 图标组件开发

## 基本信息
- **模块**: 基础组件
- **子模块**: Icon 图标组件
- **优先级**: 高
- **状态**: 待开发
- **创建时间**: 2026-04-13

## 需求描述
[详细描述]

## 功能点
1. [功能点1描述]
   - 业务规则: [规则1]; [规则2]
   - 约束条件: [约束1]
2. [功能点2描述]
   - 业务规则: [规则1]
   - 约束条件: [约束1]

## 前置条件
- [条件1]
- [条件2]

## 后置条件
- [条件1]
- [条件2]

## 业务规则
| 规则编号 | 规则描述 | 优先级 |
|---------|---------|--------|
| BR-001 | [描述] | 高 |
| BR-002 | [描述] | 中 |

## 异常处理
- **[异常场景]**: [预期处理]
- **[异常场景]**: [预期处理]

## 标签
标签1、标签2、标签3

## 依赖关系
- 前置需求: 无
- 后续需求: 无

## 变更历史
| 日期 | 操作 | 说明 | 操作人 |
|------|------|------|--------|
| 2026-04-13 | 创建 | AI 解析生成 | AI |
```

## 编号规则

### 需求ID
- 格式: `REQ-XXX` (XXX为3位数字)
- **模块内唯一**，非全局唯一
- 示例: 基础组件下可以有 REQ-001, REQ-002

### 功能点
- 不使用独立编号，用列表序号表示 (1., 2., 3.)

### 业务规则
- 格式: `BR-XXX` (XXX为3位数字)
- **需求内唯一**
- 示例: REQ-001 下可以有 BR-001, BR-002

## 关键区别

### 与之前格式的不同

| 方面 | 旧格式 | 新格式（标准） |
|------|--------|--------------|
| 文件命名 | `DRAFT-XXX_structured_requirements.md` | `REQ-XXX_需求标题.md` |
| 需求ID | 全局唯一 | **模块内唯一** |
| 功能点编号 | FP-001, FP-002 | **无编号，用列表序号** |
| 标题格式 | `### REQ-001: 标题` | `# REQ-001: 标题` |
| 元数据 | `> 📋 文档信息` | `## 基本信息` 表格 |
| 问题标注 | 内嵌在需求中 | **不内嵌，单独的问题检测** |
| 异常处理 | 表格格式 | **列表格式** `- **[场景]**: 处理` |

## AI生成Prompt要点

1. **文件命名**: `{模块名}/REQ-XXX_{标题}.md`
2. **需求标题**: `# REQ-XXX: {标题}`
3. **基本信息**: 使用列表格式，不是表格
4. **功能点**: 用列表序号，不用FP-XXX编号
5. **业务规则**: 用BR-XXX编号，表格展示
6. **异常处理**: 用列表格式 `- **[场景]**: 处理`
7. **问题检测**: 不内嵌在文档中，单独返回给前端

## 草稿处理

草稿保存为JSON格式，包含：
```json
{
  "draft_id": "DRAFT-XXXXXXXX",
  "title": "文档标题",
  "module": "模块名称",
  "source_name": "来源文件名",
  "requirements": [
    {
      "id": "REQ-001",
      "title": "需求标题",
      "module": "模块名",
      "content": "完整的Markdown内容"
    }
  ],
  "issues": [...],
  "created_at": "...",
  "updated_at": "..."
}
```

发布时：
1. 按模块创建目录
2. 每个需求单独保存为 `REQ-XXX_标题.md`
3. 删除草稿
