# 结构化需求 → 测试用例

根据结构化需求文档自动生成**完整的测试用例**，保证覆盖全、结构清晰、步骤明确。

## 工作流

1. **获取输入**：接收一个或多个结构化需求文件，提取需求详情（功能点、业务规则、前置/后置条件、异常处理）。
2. **分析需求**：
   - 识别每个需求的功能点和业务规则
   - 分析前置条件和后置条件
   - 识别异常场景和边界条件
   - 识别状态流转和依赖关系
3. **设计测试用例**：
   - **正向测试**：正常流程验证
   - **反向测试**：异常流程和错误处理
   - **边界值测试**：临界值验证
   - **等价类测试**：输入域划分（在适用时）
   - **场景法测试**：端到端业务流程
   - **状态流转测试**：状态机验证（当需求涉及状态机、多步骤流程时）
4. **生成测试用例**：按下方 [测试用例格式](#测试用例格式) 输出 JSON。
5. **交付结果**：返回测试用例列表，包含用例属性、步骤、预期结果等。

## 测试用例设计策略（必须覆盖）

### 2.1 正向测试（正常流程）

- **含义**：在合法前置条件下，按需求规定的正常路径执行，验证系统行为符合文档约定。
- **做法**：为每个核心功能点至少设计 1 条「正常流程」用例；步骤与预期与需求一致，明确「成功」的判定标准（如界面变化、状态、关键数据）。

### 2.2 反向 / 异常测试（负向用例）

- **含义**：使用非法输入、错误操作或违反约束的条件，验证系统能正确拒绝、提示或返回约定错误，且不产生副作用。
- **做法**：针对每个可校验的输入/条件，至少考虑一类无效情况（格式错误、必填缺失、越权、重复提交等）；预期结果明确（提示文案、错误码、不写库、不改变状态）。

### 2.3 边界值设计

- **含义**：在输入或条件的边界附近设计用例（最小值、最大值、刚好超界、空值、长度临界等），暴露截断、溢出等问题。
- **做法**：从需求中提取所有「有范围/有长度/有数量限制」的字段；对每个边界设计：边界内有效值、边界值、边界外无效值（若需求有定义）；对可选/可空字段考虑空串、未传等；对数值考虑 0、负值、极大值（若业务允许）。

### 2.4 等价类划分（在适用时）

- **含义**：将输入域划分为若干等价类，从每类中选取代表值设计用例，在保证覆盖的前提下减少冗余。
- **做法**：有效等价类选 1～2 个代表值；无效等价类对每种违规类型（格式、范围、必填缺失）各选代表值；可与边界值结合。

### 2.5 场景法 / 用户场景

- **含义**：以真实用户场景或业务故事为线索，串联多个功能点，形成端到端或跨模块用例。
- **做法**：从需求中归纳 2～3 个典型用户目标或业务场景；每个场景下设计一条或多条用例，覆盖主流程与常见分支；可与正向、反向、边界结合。

### 2.6 状态与流程（当需求涉及状态机、多步骤流程时）

- **含义**：针对状态流转、角色切换、多步骤业务流程设计用例，避免遗漏中间状态或非法跳转。
- **做法**：列出主要状态与允许的迁移；设计从初态到终态的正向路径、中断/回退路径、在非法状态下执行操作（应被拒绝或提示）；若有角色/权限，覆盖越权与角色切换后的可见性。

---

**重要**：未指定专项时，**不得**仅输出少量「正常流程」用例，须按本规范覆盖边界值、场景法、反向与状态等策略。

## 测试用例格式

输出必须是严格的 JSON 格式，包含以下结构：

```json
{
  "test_cases": [
    {
      "id": "TC-001",
      "title": "模块-场景描述",
      "description": "用例简要说明",
      "requirement_id": "REQ-001",
      "module": "模块名称",
      "sub_module": "子模块名称（可选，如果需求有子模块则必须填写）",
      "priority": "高/中/低",
      "case_type": "基本功能/异常情况/边界条件/性能测试/安全测试",
      "test_method": "正向测试/反向测试/边界值/场景法/状态流转",
      "preconditions": ["前置条件1"],
      "steps": [
        {
          "step_no": 1,
          "action": "操作步骤",
          "input": "输入数据（可选）",
          "expected": "预期结果"
        }
      ],
      "expected_result": "整体预期结果（总结）",
      "test_data": {
        "key1": "value1"
      },
      "tags": ["标签1", "标签2"]
    }
  ],
  "generation_summary": "生成总结（说明生成了多少用例，覆盖哪些策略）",
  "warnings": ["警告信息列表"]
}
```

## 字段说明

### 用例 ID 规则
- **格式**：`TC-XXX`（三位数字，从 001 开始递增）
- **全局唯一**：所有模块的用例 ID 不重复
- **连续递增**：TC-001, TC-002, TC-003...

### 用例标题格式
- **格式**：`{模块}-{场景描述}`
- **示例**：
  - `用户登录-手机号验证码正常流程`
  - `订单提交-必填项缺失时提示`
  - `密码修改-原密码错误`
- **要求**：一句话概括场景，简洁明了

### 优先级定义
| 级别 | 含义 | 判断标准 |
|------|------|---------|
| 高 | 核心流程，必须测试 | 缺少会导致系统无法使用 |
| 中 | 重要功能，建议测试 | 影响用户体验 |
| 低 | 辅助功能，可选测试 | 锦上添花 |

### 用例类型（case_type）
| 类型 | 说明 | 示例 |
|------|------|------|
| 基本功能 | 正常功能验证 | 登录成功、下单成功 |
| 异常情况 | 异常流程验证 | 密码错误、库存不足 |
| 边界条件 | 边界值验证 | 长度限制、数值范围 |
| 性能测试 | 性能指标验证 | 响应时间、并发数 |
| 安全测试 | 安全性验证 | 权限、注入攻击 |

### 测试方法（test_method）
| 方法 | 说明 | 适用场景 |
|------|------|---------|
| 正向测试 | 正常流程 | 主要业务流程 |
| 反向测试 | 异常流程 | 错误处理、校验 |
| 边界值 | 边界条件 | 数值、长度、范围 |
| 场景法 | 端到端流程 | 跨模块业务场景 |
| 状态流转 | 状态变化 | 订单状态、审批流程 |

### 测试步骤格式
每个步骤包含：
- **step_no**: 步骤序号（从 1 开始）
- **action**: 操作动作（动词开头，如"点击"、"输入"、"选择"）
- **input**: 输入数据（可选，如"手机号：13800138000"）
- **expected**: 该步骤的预期结果

## 用例设计原则

### 1. 完整性
- **覆盖所有功能点**：不遗漏需求中的任何功能
- **覆盖所有测试策略**：正向、反向、边界值、场景法必须有
- **异常处理验证**：需求中提到的每个异常场景都要有用例

### 2. 独立性
- 每条用例独立可执行
- 不依赖其他用例的执行结果
- 前置条件明确，可通过准备数据满足

### 3. 可追溯
- 每条用例关联到具体需求（requirement_id）
- 用例标题和描述能追溯到需求的功能点
- 便于需求变更时识别受影响的用例

### 4. 可执行性
- 步骤清晰、具体、可操作
- 预期结果明确、可验证
- 测试数据具体（避免"某值"、"适当"等模糊描述）

### 5. 粒度适中
- 不要过于细化（一步操作一条用例）
- 不要过于粗糙（一个用例包含十几个步骤）
- 建议：5-10 步/条，聚焦一个测试点

## 常见场景处理

### 场景 1：需求包含明确的功能点
- 为每个功能点设计正向、反向、边界值用例
- 业务规则单独设计用例验证

### 场景 2：需求包含状态流转
- 增加状态流转测试用例
- 覆盖正常流转、异常跳转、回退路径

### 场景 3：需求包含权限/角色
- 增加安全测试用例
- 覆盖越权访问、角色切换后的可见性

### 场景 4：需求包含性能要求
- 增加性能测试用例
- 明确负载指标和通过标准

## 输出要求

1. **严格 JSON 格式**：不要输出其他文字，只输出 JSON
2. **完整覆盖**：不遗漏需求中的任何功能点和异常场景
3. **合理拆分**：每条用例聚焦一个测试点
4. **步骤清晰**：5-10 步/条，动词开头，预期明确
5. **数据具体**：测试数据使用具体值，不要模糊描述
