# 原始 PRD → 结构化需求数据

将原始 PRD 或需求文档自动拆解为**结构化需求数据**，按功能模块拆分，每个需求包含完整的功能点、业务规则、异常处理等信息，便于后续测试用例生成和需求管理。

## 工作流

1. **获取输入**：接收用户上传的需求文档（PDF/Word/Markdown/纯文本），提取文本内容。
2. **分析需求**：
   - 识别所有功能模块和子模块
   - 为每个模块拆分为独立的需求条目
   - 提取功能点列表，标注业务规则和约束
   - 分析前置条件、后置条件
   - 提取业务规则表
   - 识别异常场景和处理方式
   - 识别需求间的依赖关系（前置/后续需求）
3. **生成结构化数据**：按下方 [结构化需求格式](#结构化需求格式) 输出 JSON。
4. **交付结果**：返回结构化需求列表，包含模块、功能点数量、警告信息等。

## 结构化需求格式

输出必须是严格的 JSON 格式，包含以下结构：

```json
{
  "modules": [
    {
      "module_name": "模块名称（中文，与需求文档一致）",
      "requirements": [
        {
          "id": "REQ-001",
          "title": "需求标题（简洁明了）",
          "sub_module": "子模块（可选）",
          "priority": "高/中/低",
          "description": "需求详细描述（2-5句话，说明功能和目标）",
          "function_points": [
            {
              "id": "FP-001",
              "description": "功能点描述（一句话说明）",
              "business_rules": ["规则1", "规则2"],
              "constraints": ["约束条件1"],
              "priority": "高/中/低"
            }
          ],
          "preconditions": ["前置条件1", "前置条件2"],
          "postconditions": ["后置条件1"],
          "business_rules": [
            {
              "id": "BR-001",
              "description": "业务规则描述",
              "priority": "高/中/低"
            }
          ],
          "exception_handling": [
            {
              "scenario": "异常场景描述",
              "expected_handling": "预期处理方式"
            }
          ],
          "prerequisite_requirements": ["前置需求ID列表，如 REQ-001"],
          "dependent_requirements": ["后续需求ID列表"],
          "tags": ["标签1", "标签2"]
        }
      ]
    }
  ],
  "parse_summary": "解析总结（2-3句话，说明识别了多少模块、功能点、需求）",
  "warnings": ["警告信息列表（如某些信息不明确、可能遗漏的内容）"]
}
```

## 字段说明

### 需求 ID 规则
- **格式**：`REQ-XXX`（三位数字，从 001 开始递增）
- **全局唯一**：所有模块的需求 ID 不重复
- **连续递增**：REQ-001, REQ-002, REQ-003...

### 功能点 ID 规则
- **格式**：`FP-XXX`（三位数字，从 001 开始递增）
- **全局唯一**：所有需求的功能点 ID 不重复

### 业务规则 ID 规则
- **格式**：`BR-XXX`（三位数字，从 001 开始递增）
- **全局唯一**：所有需求的业务规则 ID 不重复

### 优先级定义
| 级别 | 含义 | 判断标准 |
|------|------|---------|
| 高 | 核心功能，必须实现 | 缺少会导致系统无法使用 |
| 中 | 重要功能，建议实现 | 影响用户体验但不影响核心流程 |
| 低 | 辅助功能，可选实现 | 锦上添花，不影响主要功能 |

### 模块命名
- 使用**中文**，与需求文档中的用语一致
- 简洁明了，如：`用户管理`、`订单管理`、`商品管理`
- 不要使用英文或缩写

### 标签提取
从需求文档中自动提取关键词作为标签：
- 业务领域：`用户`、`订单`、`支付`
- 功能类型：`认证`、`搜索`、`通知`
- 技术特性：`安全`、`性能`、`缓存`

## 解析原则

### 1. 完整性
- **不遗漏**任何功能模块和功能点
- 从需求文档中**完整识别**所有功能、特性、用户故事
- 若有章节/目录结构，可与之对齐

### 2. 准确性
- 提取的业务规则和约束必须**与文档一致**
- 不要臆造文档中未提及的功能
- 若某些信息不明确，在 `warnings` 中标注

### 3. 结构化
- 每个需求独立、完整、可追溯
- 功能点拆解到**可测试的粒度**
- 异常处理必须具体，不要泛泛而谈

### 4. 可追溯
- 需求标题与文档中用语一致
- 功能点能追溯到原始文档的具体章节
- 依赖关系清晰（前置需求、后续需求）

## 常见场景处理

### 场景 1：需求文档结构清晰
- 直接按章节/模块拆分
- 提取标题作为需求标题
- 内容作为功能点和描述

### 场景 2：需求文档结构混乱
- 按功能块重新组织
- 合并重复内容
- 在 `warnings` 中说明重组情况

### 场景 3：需求文档信息不完整
- 提取已有信息
- 在 `warnings` 中列出缺失内容
- 标注需要用户澄清的问题

### 场景 4：需求文档包含多个产品
- 按产品线拆分模块
- 每个产品线独立分析
- 在 `warnings` 中说明多产品情况

## 输出要求

1. **严格 JSON 格式**：不要输出其他文字，只输出 JSON
2. **完整覆盖**：不遗漏需求文档中的任何功能
3. **合理拆分**：每个需求聚焦一个功能点或场景
4. **清晰描述**：功能点、业务规则、异常处理都要具体明确

## 示例

### 输入（需求文档片段）
```
# 用户管理系统

## 登录功能
用户可以通过手机号+验证码登录。
- 验证码6位，有效期5分钟
- 同一手机号60秒内只能发送一次
- 连续5次错误锁定30分钟
```
